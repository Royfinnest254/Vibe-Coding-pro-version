Anti-Slop Visual & Code Guidelines Eradicating AI Slop

Strict guardrails designed to keep codebases and interfaces professional:

  • Font Pairings: Never use generic fonts. Enforce Georgia for headings and Calibri for body text.
  • Color Architecture: Restrict CSS colors to pre-defined HSL brand tokens (Navy, Teal, Gold, Jungle Green).
  • Prohibited Elements: Zero glassmorphism (no backdrop blurs), zero gradient backgrounds, zero animated counters, zero shimmer loading screens, and zero canvas background loops.
  • Status Reports: Display statuses as uppercase text-only badges: VERIFIED, FAILED, PENDING, ERROR. No colored dots or pulse animations.
  • Code Cleanliness: Zero emojis in logs/code, zero commented-out dead code, and every query must map to a strong type (no any bindings).

Sandbox Execution Rules Test-First Isolation

To ensure system stability, every change must pass isolated verification checks in the sandbox folder:

  1. Write a prototype test script inside the scratch/ directory.
  2. Execute the script using run_command to capture the terminal stdout and log files.
  3. Inspect the execution log file at C:\Users\roych\.gemini\antigravity\scratch\.
  4. Only copy the finalized code to your active directories once the sandbox tests have been verified green.

1. Foundations & Mindsets

Setting up your baseline relationship as technical lead directing the AI developer.

Director vs. Verifier Mindset

The AI is the fastest junior developer alive: tireless, literal, and possessing zero stake in the outcome. It does exactly what you say, not what you meant, and ignores safety checks unless commanded otherwise. **If nobody acts as the engineer, your application lacks one.**

Your value is in **Direction** (specifying boundaries) and **Verification** (confirming validity).

  • Runs does not mean works: Just because a page rendered and didn't crash does not mean the business logic, security checks, and data mutations are correct.
  • No blind approvals: If a plan or code change is unclear, demand explanation before giving the green light.

2. The Build System (The 8 Disciplines)

Mandatory process habits that eliminate technical debt and application breakdown.

1. Spec Before Code

AI memory decays in long chat threads. It is prone to "logic drift"—quietly rewriting login, permissions, or database operations while building something unrelated later.

The Fix: Never start a feature without establishing a spec. Maintain the Feature Spec containing specific **acceptance criteria** ("done when" lists).

2. Prompting as Spec-Writing

Vague instructions like "make this page nicer" cause massive, uncontrolled changes. A professional prompt acts as a scoped specification with strict guardrails.

Always structure prompts with: Context, Scope of the change, Blast radius check, Constraint, and a Plan-gate approval request.

3. Git, Your Safety Net

Never write code without Git tracking. It is your ultimate "undo" button. If the agent thrashes or corrupts your codebase, rollback immediately.

Branch: main

Git Guideline: Create a feature branch for experiments. Commit every time a micro-slice passes verification. If a slice breaks everything, scrap the branch and restore from main.

4. Build in Small Slices

Deconstruct large components into the smallest possible functional slices (e.g., UI layout → mock data binding → database connection → error boundary). Validate each slice before moving to the next. This prevents hitting the "70% Done Wall" where logic breaks down entirely.

5. Security You Can't Skip

AI writes happy-path code by default, ignoring access controllers unless explicitly instructed to include them. Enforce these 5 security guards in your specifications:

1. Server-side checks — Validate access behind the API, not just in UI view bindings.
2. Secure secrets — Pass keys in `.env` configurations.
3. Input sanitization — Sanitize user form inputs against SQL Injection or XSS attacks.
4. Rate limiting — Protect APIs against spam bots.
5. Row Isolation — Ensure users can only query their own primary keys.

6. Verification & Checking

Never rely on a basic UI click-through to confirm a feature works. Test edge cases, empty form states, validation failures, and attempt to hit backend API endpoints directly using tools (curl) to verify access controls hold.

3. Driving Antigravity & Troubleshooting

Best practices for token efficiency, code reviews, and resolving application bugs.

Antigravity Specific Commands

Treat Antigravity as an agent orchestration deck. Use these habits to reduce costs and maintain precision:

  • Use Planning Mode first: Allow the agent to construct an implementation plan as an Artifact. Review it carefully before granting permission to write files.
  • Leverage Browser Automation: Ask the agent to launch test runs inside headless browsers and capture screenshots/recordings to prove correctness.
  • Set local project rules: Place a customized AGENTS.md rules file at the root of the workspace. The agent reads and adheres to these rules on every invocation.

Debugging Playbook (Diagnose Before Medicating)

When an error occurs, do not thrashes or re-prompt blindly. Follow this diagnosis pipeline:

  1. Capture the raw error log or stack trace.
  2. Isolate when and how the error occurs (reproduction steps).
  3. Ask the AI to explain the root cause *before* changing code.
  4. Verify the proposed fix fits the spec and only alters relevant files.
  5. Implement the fix, verify, and commit.

Output Red Flags to Watch For

Red Flag Shape Severity Remediation Prompt
Frontend Paywall CRITICAL `"Move that eligibility check to the server API side behind authorization middleware."`
Hardcoded Secret CRITICAL `"Extract that API key into an environment variable and load it via process.env."`
Sprawling Scope MEDIUM `"This change is touching 10 unrelated files. Explain why each edit is required or restrict the scope."`
Missing Errors MEDIUM `"Add try/catch error boundaries and display a user-friendly error message if the database query fails."`

4. The Web Coder's Dictionary

An expanded terminology guide. Use these definitions to prompt precisely and check diffs.

Glossary Index

Term Plain English Meaning & How to Use It
Full-stack An application that consists of both a frontend UI and a backend logic/database server.
Client-side (Frontend) Everything running inside the user's web browser. Easy for users to edit and bypass checks here.
Server-side (Backend) The secure server you manage. Always place database mutations, authentication checks, and business calculations here.
API (Endpoint) The path or connector that the storefront (frontend) calls to request action from the back office (backend). E.g., `POST /api/checkout`.
ORM (Prisma / Drizzle) A library that simplifies writing database queries, protecting against SQL injection syntax vulnerabilities.
Schema (Data Model) The structural blueprint of your database. Define this first to align the AI on your data structures.
Database Transaction Grouping multiple database updates so that if one fails, all fail (e.g. charging a payment card and reserving a ticket). Prevents partial data corruption.
Status Code Standard numbers returned by servers: `200` OK, `400` Bad Input, `401` Logged Out, `403` No Access, `409` Conflict (e.g., double booking), `500` Crash.
CORS Cross-Origin Resource Sharing. Security setting that specifies which web domain is allowed to send requests to your server API. Essential to configure upon deployment.
JWT (Token) JSON Web Token. A cryptographically signed security token stored in browser session memory to verify who a user is on each API call.
Password Hashing Scrambling user passwords securely using algorithms like bcrypt or Argon2 so they cannot be read back. Plaintext passwords must never enter the database.
DNS (A Record, CNAME) Domain Name System. Connects your human-readable domain to the computer's server IP addresses. Propagation delays can take up to 48 hours.
End-to-End (E2E) Test An automated test script that fires up a real browser and simulates a user clicking buttons and filling out forms, testing the complete application state.

Template: Security Audit Prompt 
Review the proposed implementation against the 5 security non-negotiables:

1. **Permission Checks**: Are authorization and permission checks performed on the server (behind the API), rather than just hiding UI components in the browser?
2. **Secrets & Keys**: Are there any API keys, passwords, credentials, or private URLs written directly in the code (especially frontend client files) instead of environment variables?
3. **Input Validation**: Is every field in every user input validated and cleaned on the server before database write or processing?
4. **Rate Limiting**: Are public or unauthenticated endpoints rate-limited to prevent abuse or bot spam?
5. **Data Isolation**: Does every database query verify record ownership, ensuring one user cannot fetch or modify another user's records simply by changing an ID parameter?

For each rule:
- Answer YES or NO.
- State exactly where in the codebase this rule is enforced (file and line numbers).
- Highlight any concerns or remediation steps if a rule is not fully satisfied.
